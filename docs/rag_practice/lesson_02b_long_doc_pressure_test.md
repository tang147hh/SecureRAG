# 实操第 2B 课：长文档 Chunk 压力测试

第 2 课中，短文档在 `300/50` 和 `1024/256` 下都能正确回答，说明短文档不足以暴露切块参数差异。

本节使用更长、更混杂的制度文档测试 chunk 参数。

## 测试文档

```text
long_policy_mixed.md
```

它包含：

```text
年假
病假
报销
特殊编号
财务权限
文档权限
生产环境访问
离职账号
例外条款
```

## 问题集

```text
long_policy_questions.csv
```

问题类型包括：

```text
事实型
条件型
精确编号型
权限型
无答案拒答型
```

## 实验设置

只对 `long_policy_mixed.md` 做测试。

先跑两组：

```text
A: chunk_size=300, chunk_overlap=50
C: chunk_size=1024, chunk_overlap=256
```

其他参数保持一致：

```text
retrieval_mode: hybrid
top_k: 10
rerank: 关闭
prompt: 默认 RAG
```

## 重点观察

### 300/50

观察它是否：

```text
更容易命中精确的小条款
引用更短、更聚焦
条件型问题是否缺上下文
```

### 1024/256

观察它是否：

```text
上下文更完整
但引用是否混入多个章节
是否把相似制度混在一起
```

## 推荐先测问题

```text
LQ002 一次申请连续 5 天以上年假需要提前多久？
LQ003 试用期员工能否在试用期内申请年假？
LQ005 超过 90 天未提交的报销单是否受理？
LQ006 BX-2048 当前是什么状态？
LQ009 付款审批模块谁可以访问？
LQ011 生产环境访问申请必须说明哪些内容？
LQ012 公司明年是否计划上市？
```

这些问题比短文档更容易暴露 chunk 大小差异。

## 判断结论

如果 `300/50` 表现更好：

```text
说明当前文档更适合细粒度检索。
后续可以测试 rerank 是否进一步提升排序。
```

如果 `1024/256` 表现更好：

```text
说明问题需要更多上下文。
后续可以测试父子块或扩大 context。
```

如果两者都很好：

```text
说明当前 embedding + hybrid search 已经足够覆盖该文档。
下一步应测试 retrieval_mode: vector / text / hybrid 的差异。
```
